# Project Plan

## 1. 최종 목표

이 저장소는 코딩 작업, Kaggle/ML 실험, 오류 수정, 코드 리뷰를 체계적으로 관리하기 위한 프로젝트 작업 공간이다.

최종 목표는 다음과 같다.

- 코딩 작업의 목표와 범위를 명확히 기록한다.
- 모델 학습/추론 실험의 재현성을 높인다.
- 오류 수정 과정을 추적 가능하게 만든다.
- 최소 수정 원칙을 유지하면서 코드 품질을 개선한다.
- 작업 완료 후 수정 기록과 테스트 결과를 남긴다.

---

## 2. 구현 범위

현재 프로젝트에서 관리할 작업 범위는 다음과 같다.

### 2.1 코딩 작업

- Python 코드 작성 및 수정
- Kaggle Notebook 코드 정리
- 학습/추론 파이프라인 개선
- 제출 파일 생성 로직 점검
- 라이브러리 버전 충돌 해결

### 2.2 머신러닝 실험

- 데이터 전처리
- train/valid split 관리
- seed 고정
- 모델 학습 설정 관리
- validation metric과 leaderboard score 차이 분석
- inference 후처리 및 threshold 튜닝

### 2.3 디버깅

- traceback 분석
- import 오류 해결
- 경로 오류 해결
- 타입/컬럼명 불일치 수정
- GPU/CPU 환경 차이 대응

### 2.4 문서화

- 작업 계획 기록
- 결정 이유 기록
- 할 일 체크리스트 관리
- 수정 기록 관리

---

이 프로젝트에서 필요한 기본 기능은 다음과 같다.

- 최소 수정 코드 제안
- 테스트 방법 작성
- 작업 완료 후 수정 기록 작성
- 남은 리스크 정리

## 5. 예상 리스크

| 리스크 | 설명 | 대응 |
|---|---|---|
| 라이브러리 버전 충돌 | Kaggle/Colab/로컬 환경마다 패키지 버전이 다를 수 있음 | 실행 환경과 버전 먼저 확인 |
| GPU 사용 불가 | 학습 시간이 길어지거나 일부 코드가 실패할 수 있음 | CPU fallback 또는 작은 샘플 테스트 적용 |
| 데이터 경로 불일치 | `/kaggle/input`, `/lambda/nfs` 등 환경별 경로가 다름 | 경로를 상단 변수로 분리 |
| 컬럼명 불일치 | train/test/submission 컬럼명이 다를 수 있음 | 로드 직후 `df.columns` 확인 |
| 리더보드 과적합 | public LB 점수만 보고 과도하게 튜닝할 위험 | validation 전략과 함께 판단 |
| 전체 재작성 위험 | 작은 오류 수정 중 구조가 과도하게 바뀔 수 있음 | 최소 수정 원칙 유지 |

## 7. 운영 원칙

모든 코딩 작업은 다음 순서로 진행한다.

1. 작업 이해
2. 작업 계획
3. 수정 또는 생성할 코드
4. 자체 점검 결과
5. 다음 작업

작업을 시작하기 전에는 목표, 수정할 파일, 예상 영향 범위, 주의할 점을 먼저 확인한다.
