# Project Context

## 1. 현재 프로젝트 구조

현재 저장소는 AI 코딩 보조 작업, Kaggle/ML 실험, 오류 수정 및 프로젝트 운영 규칙을 관리하기 위한 구조로 사용한다.

현재 포함된 주요 문서는 다음과 같다.

| 파일 | 역할 |
|---|---|
| `PROJECT_SKILLS.md` | 작업용 Superpowers / Skill 매뉴얼 |
| `PROJECT_PLAN.md` | 목표, 범위, 리스크 관리 |
| `PROJECT_CONTEXT.md` | 현재 코드 구조 및 결정 이유 기록 |
| `PROJECT_TODO.md` | 작업 체크리스트 및 테스트 관리 |

---

## 2. 현재 개발 방향

현재 프로젝트는 다음 방향을 우선 목표로 한다.

- 최소 수정 원칙 유지
- 재현 가능한 실험 환경 유지
- Kaggle/ML 중심 작업 최적화
- 오류 원인 추적 가능성 강화
- 작업 기록 문서화
- 테스트 누락 최소화

---

## 3. 이전 결정 이유

### 3.1 `PROJECT_SKILLS.md`를 분리한 이유

단순 코드 작성이 아니라 다음 작업 흐름을 표준화하기 위해 분리했다.

- 작업 목표 정리
- 영향 범위 분석
- 오류 원인 분석
- 최소 수정 코드 작성
- QA 점검
- 테스트 절차 작성
- 수정 기록 관리

이를 통해 대규모 리팩토링 없이 안정적으로 작업하기 위한 목적이다.

---

### 3.2 `.md` 확장자를 사용한 이유

문서 파일은 Markdown 형식으로 관리하는 것이 GitHub에서 읽기 쉽고 유지보수에 유리하기 때문이다.

`.m`은 MATLAB/Objective-C 파일로 해석될 가능성이 있어 문서용으로 적합하지 않다고 판단했다.

---

### 3.3 최소 수정 원칙을 사용하는 이유

현재 프로젝트는 Kaggle/ML 실험 코드와 오류 수정 작업 비중이 높다.

전체 구조를 자주 갈아엎으면:

- validation 결과 재현이 어려워지고
- leaderboard score 비교가 꼬일 수 있으며
- 이전 실험과 비교가 힘들어진다.

따라서 기존 구조를 최대한 유지하면서 필요한 부분만 수정하는 방식을 기본 원칙으로 한다.

---

## 4. 현재 사용 중인 작업 스타일

### 4.1 코딩 작업 방식

- 기존 코드 구조 우선 유지
- 함수 단위 수정 우선
- 전체 재작성 최소화
- 오류 재현 가능성 우선 확인
- 버전 충돌 우선 점검

---

### 4.2 머신러닝 실험 방식

- seed 42 기본 사용
- validation metric과 public LB를 함께 비교
- soft label 실험 시 데이터 leakage 여부 확인
- tokenizer/model 불일치 여부 점검
- inference 후처리 영향 분석

---

## 5. 참고 기준

현재 프로젝트는 다음 환경을 자주 사용한다.

| 환경 | 설명 |
|---|---|
| Kaggle Notebook | 기본 실험 환경 |
| Lambda Cloud | 대규모 학습 환경 |
| Transformers | Hugging Face 모델 학습 |
| PEFT | LoRA/parameter efficient tuning |
| Accelerate | 분산/가속 학습 |
